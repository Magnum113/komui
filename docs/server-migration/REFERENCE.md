# Полный технический справочник по переносу KOMUI

Дата подготовки: 25 июня 2026 года.

Этот файл сохраняет полный технический проект. Исполнимый порядок работ,
критерии перехода и статус этапов находятся в
[`SERVER_MIGRATION_PLAN.md`](../../SERVER_MIGRATION_PLAN.md).

## Обязательный принцип изоляции

Вся подготовка и проверка сначала выполняется в отдельном staging-контуре.
До явного решения владельца не изменяются:

- действующий Vercel deployment;
- runtime-конфигурация `komui.ru`;
- production DNS;
- production webhook Т-Банка;
- данные и схема production Supabase;
- production credentials и реальные отправления СДЭК.

Staging работает на отдельной PostgreSQL-копии и не имеет пути записи обратно
в Supabase. После полной подготовки работа останавливается для ручной проверки.
Production cutover является отдельным необязательным этапом.

## 1. Цель проекта

Перенести витрину KOMUI, API, PostgreSQL, оплату, доставку, промокоды, генерацию SEO-страниц и эксплуатационные процессы на один управляемый сервер без установки Supabase и без зависимости от Vercel.

После завершения переноса:

- сайт раздаётся Nginx непосредственно с сервера;
- данные хранятся в обычном PostgreSQL 17;
- браузер не имеет прямого доступа к базе данных;
- функции оплаты, СДЭК и промокодов выполняет собственный backend;
- Supabase REST API, Edge Functions, Auth, Storage и ключи Supabase не используются;
- Vercel Functions и Vercel deploy hook не используются;
- обновление приложения, резервное копирование и восстановление контролируются владельцем сервера.

Внешними останутся только необходимые бизнес-сервисы:

- Т-Банк;
- СДЭК;
- Яндекс Карты;
- Ozon API, если он используется внешней админкой или синхронизацией;
- DNS-провайдер и центр сертификации TLS.

Для полной независимости от CDN также необходимо перенести изображения с `ir.ozone.ru` и шрифты Google Fonts на свой сервер.

## 3. Целевая архитектура

Для сервера с 2 vCPU, 4 ГБ RAM и ограниченным диском рекомендуется нативная установка без Docker. Она требует меньше памяти и диска и проще для текущего небольшого проекта.

```text
Интернет
   |
   v
Nginx :443
   |
   +-- komui.ru
   |     +-- статические HTML/CSS/JS
   |     +-- страницы товаров и коллекций
   |     +-- локальные изображения и шрифты
   |
   +-- api.komui.ru или komui.ru/api
         |
         v
   Node.js backend на 127.0.0.1:3000
         |
         +-- PostgreSQL 17 на 127.0.0.1:5432
         +-- API Т-Банка
         +-- API СДЭК
         +-- API Ozon при необходимости
```

### 3.1. Компоненты

#### Nginx

Назначение:

- HTTPS;
- раздача статики;
- reverse proxy для backend;
- лимиты размера запросов;
- базовый rate limiting;
- security headers;
- кэширование статических файлов;
- access/error logs с ограниченной ротацией.

#### Backend

Рекомендуемый стек:

- активная LTS-версия Node.js;
- TypeScript;
- Fastify;
- `pg` для PostgreSQL;
- `zod` или JSON Schema для валидации;
- `pino` для структурированных логов;
- `@fastify/rate-limit`;
- `@fastify/helmet`;
- `@fastify/cors`.

Причины выбора:

- небольшой расход памяти;
- зрелый PostgreSQL-драйвер;
- простое перенесение существующей TypeScript-логики;
- один процесс заменяет 6 Edge Functions и 2 Vercel Functions;
- штатный запуск через systemd.

#### PostgreSQL

- PostgreSQL 17, чтобы не выполнять миграцию между major-версиями;
- доступ только с localhost или Unix socket;
- отдельные роли для миграций, приложения и backup;
- без PostgREST и прямого браузерного доступа;
- регулярный `VACUUM`/`ANALYZE`;
- контролируемое хранение WAL и логов.

#### systemd

Используется для:

- автоматического запуска backend после перезагрузки;
- перезапуска при падении;
- ограничения ресурсов;
- загрузки переменных из защищённого env-файла;
- выполнения backup и SEO rebuild по timer units.

## 4. Новая модель безопасности

### 4.1. База данных

База не должна быть доступна из интернета.

Роли:

- `komui_owner` — владелец схемы, без удалённого входа приложения;
- `komui_migrator` — применение миграций;
- `komui_app` — минимальные права backend;
- `komui_backup` — только чтение для backup.

Роль `komui_app` получает доступ только к необходимым таблицам и последовательностям. Backend не должен подключаться суперпользователем.

Supabase-специфичные роли, grants и политики `anon`, `authenticated`, `service_role` после восстановления должны быть удалены или заменены.

RLS можно сохранить как дополнительный слой, но она больше не является основной границей безопасности. Основная граница — отсутствие публичного SQL/REST-доступа и минимальные права роли приложения.

### 4.2. Известная уязвимость текущего проекта

В действующем Supabase 18 внутренних таблиц публично разрешают чтение и запись через `anon`/`public`. Среди них склад, транзакции, Ozon и финансовые операции.

До публичного запуска нового сервера необходимо:

1. Закрыть эти права в текущем Supabase.
2. Не переносить политики `using (true) with check (true)` без анализа.
3. Не выдавать браузеру никаких ключей, способных писать в БД.
4. Выполнять административные изменения только через защищённый backend.

### 4.3. Секреты

Секреты хранятся вне репозитория:

```text
/etc/komui/backend.env
```

Права:

```text
owner: root
group: komui
mode: 640
```

Пример состава без реальных значений:

```dotenv
NODE_ENV=production
PORT=3000
DATABASE_URL=postgresql://komui_app:...@127.0.0.1:5432/komui
PUBLIC_SITE_URL=https://komui.ru
PUBLIC_API_URL=https://api.komui.ru

TBANK_TERMINAL_KEY=...
TBANK_PASSWORD=<set-on-server>
TBANK_API_URL=https://securepay.tinkoff.ru/v2
TBANK_TAXATION=...
TBANK_TAX=...

CDEK_LOGIN=...
CDEK_PASSWORD=<set-on-server>
CDEK_API_BASE_URL=https://api.cdek.ru
CDEK_SHIPMENT_POINT=...
CDEK_SENDER_NAME=...
CDEK_SENDER_PHONE=...

YANDEX_MAPS_API_KEY=...
OZON_CLIENT_ID=...
OZON_API_KEY=<set-on-server>
```

Запрещено:

- хранить секреты в Git;
- передавать DB password или секреты Т-Банка в браузер;
- логировать Authorization headers;
- логировать телефон, полные ФИО и платёжные payload без маскирования.

### 4.4. Сетевая защита

- открыть наружу только 80 и 443;
- SSH разрешить по ключам, по возможности только с доверенных IP;
- запретить наружу 3000 и 5432;
- отключить password login и root login по SSH после проверки ключевого доступа;
- включить UFW или nftables;
- установить fail2ban либо аналог;
- включить автоматические security updates;
- настроить TLS через Let's Encrypt;
- установить лимиты запросов в Nginx и backend.

### 5.2. Соответствие текущим функциям

| Сейчас | После переноса |
|---|---|
| `cdek-delivery-points` | `POST /v1/delivery/points` |
| `cdek-delivery-quote` | `POST /v1/delivery/quote` |
| `promo-validate` | `POST /v1/promos/validate` |
| `tbank-create-payment` | `POST /v1/payments` |
| `tbank-payment-status` | `POST /v1/payments/status` |
| `tbank-webhook` | `POST /v1/webhooks/tbank` |
| Supabase REST storefront | `GET /v1/products` |

### 5.3. Совместимый переходный слой

Чтобы снизить риск переключения, на первом этапе backend должен временно понимать старые маршруты:

```text
POST /api/supabase-function?name=cdek-delivery-points
POST /api/supabase-function?name=cdek-delivery-quote
POST /api/supabase-function?name=promo-validate
POST /api/supabase-function?name=tbank-create-payment
POST /api/supabase-function?name=tbank-payment-status
```

Совместимый слой возвращает тот же JSON, что и текущие функции. После успешного запуска frontend переводится на чистые `/v1/*` endpoints, а совместимый слой удаляется.

Такой порядок позволяет отделить перенос инфраструктуры от переработки frontend.

### 5.4. Требования к API

Каждый endpoint должен:

- принимать только заявленный HTTP-метод;
- иметь строгую схему входных данных;
- ограничивать длину строк и размер body;
- не доверять цене, скидке и стоимости доставки из браузера;
- повторно получать цену товара из PostgreSQL;
- быть идемпотентным для создания платежа;
- не отдавать внутренние SQL-ошибки клиенту;
- иметь request ID;
- маскировать персональные данные в логах.

Origin/CORS не является аутентификацией. Публичные endpoints дополнительно защищаются rate limiting, валидацией и при необходимости CAPTCHA.

## 6. Перенос бизнес-логики

### 6.1. Общий подход

Существующие Edge Functions не следует переписывать с нуля. Нужно:

1. Перенести чистые функции расчёта и валидации без изменений.
2. Заменить `Deno.env.get()` на централизованный config module.
3. Заменить `supabase-js` на repository-слой с параметризованными SQL-запросами.
4. Заменить `Deno.serve()` на Fastify routes.
5. Сохранить существующие проверки подписи Т-Банка.
6. Сохранить идемпотентность через `client_request_id`, `order_number` и уникальные индексы.
7. Добавить транзакции PostgreSQL там, где создаются заказ, позиции, попытка платежа и резерв промокода.

### 6.3. PostgreSQL repository-слой

Все SQL-запросы:

- параметризованные;
- хранятся в repository-модулях;
- не собирают SQL из пользовательских строк;
- выполняются с timeout;
- используют транзакции для связанных изменений;
- имеют тесты на happy path и ошибки.

Для создания заказа рекомендуется одна транзакция:

1. Проверить или заблокировать idempotency key.
2. Получить активные товары и цены.
3. Рассчитать subtotal.
4. Проверить промокод.
5. Проверить СДЭК и стоимость доставки.
6. Создать заказ.
7. Создать позиции.
8. Зарезервировать промокод.
9. Создать payment attempt.
10. Зафиксировать транзакцию.
11. Вызвать Т-Банк.
12. Обновить результат попытки отдельной транзакцией.

Вызов внешнего API нельзя держать внутри долгой PostgreSQL-транзакции.

### 6.4. Т-Банк

Обязательно сохранить:

- серверный пересчёт суммы;
- формирование token/signature;
- проверку подписи webhook;
- сравнение суммы webhook с ожидаемой;
- уникальность события webhook;
- идемпотентную обработку повторных уведомлений;
- перевод заказа в `paid` только после валидного webhook;
- отсутствие доверия к redirect пользователя.

Новый callback:

```text
https://api.komui.ru/v1/webhooks/tbank
```

URL должен формироваться из `PUBLIC_API_URL`, а не из `SUPABASE_URL`.

### 6.5. СДЭК

Перенести:

- OAuth token cache;
- поиск города;
- поиск ПВЗ;
- серверный расчёт тарифа;
- проверку выбранного ПВЗ;
- создание отправления после подтверждённой оплаты;
- сохранение request/response;
- повторную безопасную синхронизацию статуса.

Необходимо определить timeout, retry и circuit breaker:

- короткий timeout для UI-запросов;
- ограниченное число повторов только для безопасных idempotent GET;
- отсутствие слепого повтора создания отправления;
- уникальность отправления по `order_id`.

### 6.6. Промокоды

Сохранить:

- нормализацию кода;
- временные окна;
- глобальный лимит;
- лимит на телефон;
- резервирование;
- погашение после оплаты;
- освобождение после ошибки или отмены;
- защиту от параллельного превышения лимита.

Проверку и резервирование лимитов желательно выполнять с блокировкой строки промокода или подходящим уровнем транзакционной изоляции.

## 7. Перенос базы данных

### 7.1. Почему нельзя использовать только локальные миграции

В Supabase применена 31 миграция, а в этом репозитории есть только 5 файлов. Отсутствуют исходные миграции склада, производства, Ozon, финансов и каталога.

Нужен полный логический dump работающей PostgreSQL 17.

### 7.2. Подготовка dump

Перед выполнением команд необходимо проверить синтаксис установленной версии `pg_dump` и Supabase CLI через `--help`.

Для переноса в обычный PostgreSQL предпочтителен PostgreSQL 17 client и dump пользовательских схем:

```text
public
private
```

Dump должен включать:

- таблицы;
- данные;
- sequences/identity;
- индексы;
- foreign keys;
- функции;
- триггеры;
- comments;
- необходимые extensions.

Не нужно переносить:

- пустую Supabase Auth-инфраструктуру;
- пустой Supabase Storage;
- Realtime;
- Supabase системные роли;
- Supabase API gateway;
- Vercel deploy webhook.

### 7.3. Особые объекты

В целевой базе необходимо оставить или установить:

- `pgcrypto`;
- PL/pgSQL;
- UUID-функции, которые реально используются.

Не переносить как рабочую зависимость:

- `pg_net`;
- `notify_vercel_storefront_changed()`;
- trigger `storefront_products_redeploy`.

Вместо Vercel webhook используется локальный механизм rebuild.

### 7.4. Очистка после restore

После тестового restore:

1. Удалить Supabase-специфичные grants.
2. Удалить опасные публичные политики.
3. Удалить Vercel webhook function и trigger.
4. Создать роли приложения.
5. Назначить владельцев объектов.
6. Выдать минимальные grants.
7. Проверить sequence values.
8. Выполнить `VACUUM ANALYZE`.
9. Сравнить row counts.
10. Выполнить прикладные контрольные запросы.

### 7.5. Контроль переноса

Сравниваются минимум:

- список всех 29 таблиц;
- количество строк по каждой таблице;
- количество активных storefront products;
- суммы и количество Ozon financial operations;
- количество заказов и payment attempts;
- foreign key violations;
- значения identity sequences;
- наличие функций и триггеров;
- выборочные контрольные записи;
- хэши экспортированных ключевых наборов без персональных данных.

## 8. Переработка frontend и генерации страниц

### 8.1. Удаление Supabase из runtime

Нужно изменить:

- `data/api-config.js`;
- запрос каталога в `index.html`;
- запросы checkout;
- запрос payment status;
- `scripts/build-products.js`;
- `api/supabase-function.js`;
- `api/delivery-config.js`;
- документацию и env-примеры.

Целевое состояние:

- нет runtime-запросов к `*.supabase.co`;
- нет publishable/anon key;
- нет `functionsProxyUrl`;
- frontend знает только публичный API URL;
- секрет Яндекс Карт выдаётся через контролируемый config endpoint или статический публичный config.

### 8.2. Каталог

Frontend получает каталог:

```text
GET https://api.komui.ru/v1/products
```

Ответ должен содержать только публичные поля. Внутренние себестоимости, складские данные, Ozon raw payload и служебные metadata не должны попадать в браузер.

Локальный fallback-каталог можно оставить для отказоустойчивости, но он должен регулярно синхронизироваться.

### 8.3. SEO rebuild

Сейчас изменение каталога вызывает Vercel deploy hook. На сервере рекомендуется:

1. Все изменения каталога проводить через backend.
2. После успешной транзакции создавать durable rebuild job в PostgreSQL.
3. Worker или systemd timer забирает job.
4. `scripts/build-products.js` получает данные из локального API.
5. Генерация выполняется во временной директории.
6. После проверки файлы переключаются атомарно.
7. Повторные изменения объединяются в один rebuild.

Не следует запускать shell-команду напрямую из SQL trigger.

### 8.4. Собственные изображения и шрифты

В проекте используется не менее 132 уникальных изображений с Ozon CDN.

Для полного переноса:

1. Получить список URL из БД и статических данных.
2. Скачать изображения с контролем HTTP status и MIME type.
3. Проверить размер и декодирование.
4. Назвать файлы по content hash или стабильному product/position key.
5. Сохранить mapping старый URL → локальный путь.
6. Обновить БД и generated pages.
7. Отдавать изображения через Nginx с долгим immutable cache.
8. Оставить оригинальные URL в архиве для трассировки.

Google Fonts желательно скачать и обслуживать локально. Яндекс Карты, Т-Банк и СДЭК останутся внешними по своей природе.

## 9. Подготовка сервера

### 9.1. Ограничения текущего сервера

Характеристики:

- Ubuntu 24.04 LTS;
- 2 vCPU;
- 4 ГБ RAM;
- около 2,6 ГБ доступной RAM;
- 20 ГБ SSD;
- около 9,6 ГБ свободно.

Для нативного Nginx + Node.js + PostgreSQL текущих CPU и RAM достаточно при текущем объёме магазина.

Диск остаётся главным ограничением. Нужно:

- не копировать `.git` в production release без необходимости;
- ограничить journald;
- настроить logrotate;
- не хранить неограниченное число releases;
- не хранить единственную резервную копию на этом же диске;
- контролировать рост PostgreSQL и изображений;
- сохранить минимум 3–4 ГБ свободного места.

Рекомендуется увеличить SSD до 40 ГБ и более. Если это невозможно, перенос допустим только с жёсткими лимитами логов и внешними backup.

### 9.2. Swap

Создать 2 ГБ swap для защиты от кратковременного пика памяти. Swap не заменяет RAM и не должен использоваться постоянно.

После запуска следует контролировать:

- `available memory`;
- swap usage;
- PostgreSQL connections;
- RSS backend;
- OOM events.

### 9.3. Рекомендуемая файловая структура

```text
/opt/komui/
├── releases/
│   └── <timestamp-or-commit>/
├── current -> releases/<active>
└── shared/

/var/www/komui/
└── current -> /opt/komui/current/public

/etc/komui/
├── backend.env
└── nginx/

/var/lib/komui/
├── jobs/
└── state/

/var/backups/komui/
└── local/
```

Хранить не более 3 последних application releases.

### 10.1. Начальный вариант

На первом этапе использовать контролируемый deploy script:

## 11. Резервное копирование и восстановление

### 11.1. Правило

Backup на том же SSD не является полноценной резервной копией.

Нужно минимум:

- ежедневный локальный dump;
- ежедневная отправка зашифрованного dump на другой сервер или object storage;
- еженедельная полная копия;
- ежемесячная долгосрочная копия;
- регулярная проверка восстановления.

### 11.2. Retention

Начальная политика:

- 7 ежедневных;
- 4 еженедельных;
- 6 ежемесячных.

При текущем размере БД это не создаст значительной нагрузки, но политика должна учитывать дальнейший рост Ozon raw payload и payment events.

### 11.3. Что сохранять

- PostgreSQL schema и data;
- backend migrations;
- конфигурацию Nginx без private keys;
- список необходимых env variables без значений;
- локальные изображения;
- generated pages;
- deploy scripts;
- документированный порядок восстановления.

### 11.4. Проверка restore

Не реже одного раза в месяц:

1. Развернуть пустую тестовую БД.
2. Восстановить последний backup.
3. Проверить таблицы и row counts.
4. Запустить backend на тестовой БД.
5. Проверить каталог и создание тестового заказа без реальной оплаты.
6. Зафиксировать результат проверки.

## 15. Rollback

### 15.1. До появления новых заказов

Если новый сервер не прошёл smoke checks:

- не включать checkout;
- вернуть DNS на старую инфраструктуру;
- восстановить прежний frontend config;
- оставить target для исправления.

### 15.2. После появления новых заказов

Нельзя просто вернуть DNS на старую БД: это создаст split-brain и потерю новых заказов.

Порядок:

1. Отключить checkout.
2. Зафиксировать время rollback.
3. Экспортировать новые заказы, payment attempts, events, promo redemptions и CDEK shipments из target.
4. Сверить их с Т-Банком.
5. Импортировать или вручную согласовать данные в старой системе.
6. Только после этого переключать DNS.

Dual-write между Supabase и новой БД не рекомендуется: он существенно усложняет оплату и идемпотентность. Лучше провести короткое контролируемое maintenance window и качественную репетицию.

## 16. Критерии готовности

Перенос завершён только если:

- в runtime-коде нет запросов к `supabase.co`;
- frontend не содержит Supabase key;
- каталог загружается из собственного API;
- все 29 таблиц и данные проверены;
- админка и Ozon sync переведены;
- payment create/status/webhook работают;
- CDEK points/quote/shipment работают;
- промокоды работают при параллельных запросах;
- сервер переживает reboot;
- PostgreSQL недоступен извне;
- backup автоматически создаётся и уходит наружу;
- выполнен успешный тест restore;
- логи не содержат секреты и полные персональные данные;
- disk, RAM и сервисы мониторятся;
- есть проверенный rollback runbook;
- Supabase и Vercel можно отключить без потери функций.
