# Этап 0. Безопасный доступ и фиксация исходного состояния

## Цель

Подтвердить безопасный SSH-доступ к серверу, не изменяя его состояние, и получить исходные данные для дальнейших решений.

## Входные условия

- Сервер: `89.111.152.112`.
- SSH-порт: `22`.
- Пользователь: `codex-migrate`.
- Ожидаемый ED25519 fingerprint:
  `SHA256:rky01ssE20UedgLjN5A4Q3etCXo5t6+Pv3nckIZkHng`.
- Временный публичный ключ уже добавлен.

## Действия

### 0.1. Проверка идентичности сервера

- Получить host key через `ssh-keyscan`.
- Сравнить fingerprint с предоставленным значением.
- Добавить только подтверждённый ключ в отдельный `known_hosts`.
- Проверить вход под `codex-migrate`.
- Проверить `sudo -n true`.

### 0.2. Read-only аудит

Собрать:

- версию Ubuntu и kernel;
- CPU, RAM, swap;
- файловые системы, свободное место и inode;
- список открытых портов;
- активные systemd services;
- установленные Nginx/Apache/PostgreSQL/Node.js/Docker;
- firewall;
- SSH-конфигурацию без вывода секретов;
- текущие cron/systemd timers;
- процессы с наибольшим потреблением ресурсов;
- список пользователей и sudoers в безопасном объёме;
- состояние обновлений безопасности.

### 0.3. Проверка конфликтов

Определить:

- что уже занимает 80/443/3000/5432;
- какие сайты уже размещены;
- есть ли действующая база;
- какие каталоги нельзя затрагивать;
- какие сервисы используют текущие 1,3 ГБ RAM и диск.

### 0.4. Фиксация состояния

- Сохранить локальный отчёт аудита без секретов.
- Зафиксировать `df`, `free`, `ss`, `systemctl`.
- Не устанавливать пакеты.
- Не изменять firewall, SSH, DNS и сервисы.

## Проверки

- SSH host fingerprint совпадает.
- Вход по ключу работает.
- `sudo` работает без запроса пароля.
- На сервере не выполнено ни одной изменяющей команды.
- Понятно, какие ресурсы и порты доступны.

## Результат

Отчёт о текущем состоянии сервера и перечень конфликтов/ограничений.

## GO

Переход разрешён, если сервер идентифицирован, доступ устойчив и обнаруженные сервисы не делают миграцию опасной.

## NO-GO

- fingerprint не совпадает;
- sudo не работает;
- диск критически заполнен;
- неизвестные production-сервисы используют необходимые порты;
- обнаружены признаки компрометации;
- нет возможности сделать backup существующего состояния.

## Rollback

Не требуется: этап read-only. При ошибке удалить локальную запись host key и прекратить подключение.
