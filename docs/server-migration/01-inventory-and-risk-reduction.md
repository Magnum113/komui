# Этап 1. Инвентаризация и план снижения текущих рисков

## Цель

Понять весь контур, использующий Supabase, и подготовить безопасный план hardening без риска сломать действующую реализацию.

## Зависимости

- Этап 0 завершён со статусом `GO`.
- Есть доступ к репозиториям и текущему Supabase.

## Действия

### 1.1. Карта систем

Найти:

- админку;
- Ozon sync;
- импорт финансов;
- cron и фоновые задачи;
- аналитические скрипты;
- локальные инструменты;
- другие серверы и репозитории.

Искать:

- `bkxpzfnglihxpbnhtjjq`;
- `supabase.co`;
- Supabase publishable/anon и secret/service-role variables;
- таблицы `merch_*`;
- Vercel deploy hook;
- webhook Т-Банка.

### 1.2. Матрица потребителей

Для каждого клиента зафиксировать:

| Клиент | Владелец | Читает | Пишет | Частота | Новый интерфейс |
|---|---|---|---|---|---|

Нельзя считать проект перенесённым, если есть неизвестный писатель в БД.

### 1.3. Подготовка снижения риска

В текущем Supabase проверить:

- публичную запись в 18 внутренних таблиц;
- публичное чтение финансовых/складских данных;
- grants для `anon`, `authenticated`, `public`;
- RLS policies `using (true) with check (true)`;
- доступ к backup-таблицам;
- открытый Vercel deploy hook.

Подготовить, но не применять автоматически:

- экспортировать policies и grants;
- сделать backup;
- подготовить обратный SQL;
- подготовить forward SQL hardening;
- определить тесты совместимости;
- проверить, какие существующие клиенты зависят от небезопасных прав.

Любое изменение production policies/grants выполняется только как отдельная
операция после явного подтверждения владельца. Подготовка staging не требует
изменения production Supabase.

### 1.4. Реестр секретов

Без значений зафиксировать:

- имя секрета;
- где используется;
- кто владелец;
- можно ли ротировать;
- дата последней ротации;
- куда переносится.

## Проверки

- Все известные клиенты внесены в матрицу.
- Для каждой таблицы известен источник записи.
- Подготовлены forward и rollback SQL.
- Определены проверки витрины, checkout и внешних клиентов.
- Ни один SQL hardening не применён без отдельного подтверждения.

## Результат

- Полная dependency map.
- Проверенный план закрытия критической уязвимости.
- Согласованный список переносимых функций.
- Реестр секретов без значений.

## GO

Переход разрешён, если нет неизвестных потребителей и hardening можно будет применить отдельно и обратимо. Для staging фактическое изменение production прав не требуется.

## NO-GO

- найдены клиенты без исходного кода или владельца;
- отключение публичной записи ломает бизнес-процесс;
- нет полного backup;
- невозможно определить источник изменения критичных таблиц.

## Rollback

Если production не изменялся, rollback не требуется. Если hardening был отдельно одобрен и применён, использовать заранее проверенный обратный SQL только для подтверждённо сломанного клиента.
